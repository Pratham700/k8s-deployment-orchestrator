# AI Interaction Log

The assessment requires both a record of the AI collaboration and a note on what was *directed* vs
*delegated*. This file is that summary. The **full raw transcript** of the build session is included
alongside it — see [§ Full transcript](#full-transcript).

Tool used: **Claude Code** (agentic CLI) with the Opus model, driving the editor, shell, tests, and a
browser-preview tool for visual verification.

## How the work was directed

I (the candidate) set the constraints and made the product/architecture calls; the AI did the
mechanical build-out and verification under those calls.

**Decisions I made (directed):**

- **Scope & stack.** Light frontend (I'm infra/backend-leaning), TypeScript everywhere, pnpm +
  Nx monorepo, Node 22 / Next 15 / React 19, a Kubernetes-flavoured use case, "simple but
  effective — don't over-engineer." Repo to live on my **personal** GitHub, not my employer's.
- **Use-case selection.** Presented with three options (deployment orchestrator, drift detection,
  env provisioning), I chose the **deployment orchestrator** because the workflow/failure/rollback
  shape best fits the rubric and my background.
- **Cluster approach.** I chose a **simulated** cluster over a real `kind` cluster so evaluators
  need zero setup — the AI had laid out the trade-off explicitly before I picked.

**Things I delegated (with review):**

- Monorepo scaffolding (Nx/pnpm/tsconfig wiring), the engine + simulated-cluster implementation,
  the Hono API, the Next UI, tests, and the docs — all generated by the AI, then reviewed and
  steered by me.

## How it was iterated / course-corrected

The session shows real iteration, not one-shot generation. Representative examples:

- **Module resolution.** The AI initially used `.js` import extensions; I had it standardise on
  extensionless imports after reasoning about portability across `tsx`, Vitest, and `tsc` bundler
  resolution.
- **A convoluted helper.** The first `isProgressing()` in the cluster came out as a nested ternary;
  it was rewritten into a clear `switch` before moving on.
- **pnpm 10 build scripts.** Native postinstalls (esbuild/sharp/nx) were blocked by default; fixed by
  pinning `onlyBuiltDependencies` rather than disabling the safety globally.
- **React 19 / Next types.** `React.FormEvent` tripped deprecation warnings; the handler was
  refactored to drop the typed event entirely.
- **Lint cleanup.** ESLint surfaced a redundant `eslint-disable` and the generated `next-env.d.ts`;
  both were addressed (config ignore + directive removal) rather than suppressed.

## How correctness was verified (not just "looks done")

- **Unit tests** for the engine (happy path, all failure modes, rollback-to-previous) and **API
  tests** via `app.request()` — all run on injected fast timing for determinism.
- **A live SSE smoke test** with `curl` confirmed the streamed state transitions end-to-end.
- **Visual verification** in a real browser: drove a healthy deploy (revision 1 promoted), then a
  `crash-loop` deploy, and confirmed the UI showed the failure and the **rollback to revision 1**.
- **`pnpm verify`** (typecheck + test + build across all projects) was green before finalising.
